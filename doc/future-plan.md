# Future Plan

This document is a living plan for the application after the initial JSONL
visualizer is stable. It is not a commitment that every item below is
implemented now.

The long-term direction is a source-agnostic workspace for agent histories:
multiple imported histories can be compared on one canvas, any node can become
the start of a continuation, and future parsers can support formats beyond
JSONL without changing the core graph model.

## Current Shape

The current codebase is centered on one parsed `Trajectory` at a time:

- `src-tauri/src/parser/mod.rs` exposes `parse_file(path) -> Trajectory`
- `src/App.svelte` stores one active `Trajectory`
- Dots, Bricks, keyboard navigation, and the inspector all consume that single
  trajectory
- `Trajectory` has no source identity, file identity, import metadata, or
  workspace-level relationships

That design is simple and useful for the first version, but it becomes limiting
for comparison, continuation, and non-JSONL inputs. The future architecture
should add a layer above `Trajectory` rather than making the IR source-specific.

## Product Direction

Short term:

- Import multiple history files into one visualization canvas
- Preserve each file as a distinct source with a visible label and color
- Compare runs without merging away source identity
- Keep Dots, Bricks, inspector, selection, and keyboard navigation on one
  shared ordering model

Middle term:

- Start a new continuation from any message or block
- Build a prompt/context preview from the selected node's ancestors
- Export continuations as new histories linked to the source node
- Keep original imported histories immutable

Long term:

- Support history formats beyond JSONL
- Treat JSONL as one input encoding, not the product boundary
- Add importer adapters for directories, databases, archives, APIs, and future
  agent runtime formats
- Keep the universal IR stable while parser and import layers grow by addition

## Architecture Direction

### Add A Workspace Layer

Introduce a `Workspace` above `Trajectory`.

Possible shape:

```text
Workspace
  sources: HistorySource[]
  trajectories: Trajectory[]
  links: WorkspaceLink[]
  active_view: ViewState
```

Each imported source should carry metadata outside the IR:

```text
HistorySource
  source_id
  display_name
  file_path or origin
  detected_format
  imported_at
  color_key
  parser_warnings
```

This keeps `Trajectory` source-agnostic while giving the UI enough information
to render multiple files together.

### Namespace Runtime IDs

When several files are rendered together, source histories may reuse IDs. The
workspace layer should derive render IDs such as:

```text
source_id::message_id
source_id::block_id
```

The original IDs should remain visible in the inspector. The IR should not be
changed just to avoid cross-file render collisions.

### Keep Original Histories Immutable

Imported histories should be read-only records. Comparison annotations,
continuations, and user-created links should live in the workspace layer.

This avoids confusing imported source data with data generated by the
application.

### Split Import From Parse

Rename the backend concept from "parse a file" to "import a history".

Possible backend layers:

```text
InputReader      reads bytes/text/records from a file, folder, archive, API, or database
FormatDetector   identifies the source format from sample records or metadata
Parser           converts records into Trajectory
Validator        checks IR invariants and produces warnings/orphans
Importer         returns ImportedHistory with source metadata and trajectory
```

JSONL support then becomes one `InputReader`, not a permanent assumption.

## Short-Term Plan: Multi-File Comparison

Goal: users can add several JSONL files to one canvas and compare them.

Suggested steps:

1. Add a frontend `WorkspaceState` with `sources`, `trajectories`, and
   `selectedSourceIds`.
2. Change "open file" into "add file" while keeping "open folder" for browsing.
3. Assign a stable `source_id` to each imported file.
4. Derive render-safe IDs by namespacing source IDs with message and block IDs.
5. Add source badges in Dots, Bricks, and NodeDetail.
6. Add source toggles so users can hide or isolate histories.
7. Add a source list/sidebar with file name, format, message count, warning
   count, and orphan count.
8. Keep per-source parser warnings visible.
9. Add a merged chronological view only when timestamps are present and
   trustworthy.
10. Add a source-grouped view that keeps each file in its own lane.
11. Add Dots swimlanes for comparison mode.
12. Add Bricks grouping by source for comparison mode.
13. Update keyboard navigation to traverse visible workspace items, not just one
    trajectory.
14. Update inspector copy actions to include source identity when useful.
15. Add tests for duplicate IDs across different sources.

The first comparison version should not try to infer semantic equivalence
between runs. It should make multiple histories visible together and keep their
provenance clear.

## Middle-Term Plan: Continue From Any Node

Goal: users can select a node and create a new continuation from that point.

Suggested model:

- A continuation starts from a selected message or block
- The application gathers the ancestor chain needed to reconstruct context
- The user can preview and edit the generated continuation prompt
- Provider-specific adapters submit the continuation to a model or agent
- The response is imported as a new `Trajectory`
- The workspace records a link from the source node to the new trajectory root

Important constraints:

- Do not mutate the original imported history
- Store continuation provenance outside `Trajectory`
- Make generated histories visually distinct from imported histories
- Keep provider-specific request details out of the IR
- Support dry-run/export before live provider execution

Possible continuation metadata:

```text
ContinuationLink
  from_source_id
  from_node_id
  to_source_id
  provider
  created_at
  prompt_preview
```

## Long-Term Plan: Format-Agnostic History Support

Goal: support many agentic history formats without redesigning the graph.

Input formats may include:

- JSONL transcripts
- Single JSON exports
- SQLite databases
- Directory trees with message artifacts
- Compressed archives
- Remote API exports
- Browser or desktop app export bundles
- Future agent runtime formats

The parser interface should accept an abstract record stream or document model,
not only a path to a JSONL file.

Format support should grow by adding:

- a detector
- an input reader when needed
- a parser
- fixtures
- preservation tests
- documentation of known limitations

The IR should remain stable unless a genuinely universal concept is missing.

## UI Concepts

Comparison canvas:

- Source lanes in Dots
- Source-grouped Bricks
- Per-source color accents
- Visible warning/orphan counts
- Source filters
- Search across all visible histories
- "Show only ancestors", "show descendants", and "show related tool calls"

Continuation workflow:

- Select a node
- Click continue
- Preview context
- Choose provider/runtime
- Run or export
- Import result as a linked continuation

Inspector:

- Show source metadata
- Show original ID and render-safe ID
- Show parser warnings related to the selected record
- Show continuation links in and out
- Keep raw source data available for debugging

## Testing Strategy

Add tests for:

- Multiple imported files with colliding original IDs
- Namespaced render IDs remain unique
- Per-source ordering remains stable
- Merged ordering respects timestamps and source tie-breakers
- Source filters do not break selection traversal
- Parser warnings remain attached to the correct source
- Continuation links survive workspace save/load
- Unknown future formats return `UnknownFormat` instead of guessing
- Non-JSONL input readers preserve visible content
- Imported histories remain immutable after continuations are added

## Documentation Strategy

Future docs should distinguish:

- IR design: universal graph representation
- Import design: source detection, readers, and parsers
- Workspace design: multi-source comparison and continuation links
- Renderer design: how views consume ordered workspace items
- Provider design: continuation adapters and privacy boundaries

This separation keeps the project from mixing source-specific parsing details
into the IR contract.

## Open Questions

- Should the workspace be saved as a project file?
- Should comparison mode default to lanes or merged chronology?
- Should source colors be user-editable or deterministic?
- Should continuation providers run inside the desktop app, an external CLI, or
  a plugin system?
- Should generated continuations be stored as files, workspace records, or both?
- How much raw input should be retained for very large histories?
- What is the right public name before the first release?
